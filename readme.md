# PACE: Prompt Accessibility Controlled Evaluation


PACE is a benchmarking framework for evaluating the accessibility of HTML code generated by large language models (LLMs).



The system systematically varies prompting conditions and evaluates the resulting interface code using two complementary approaches:



\- Schema-based accessibility evaluation implemented in `bench/scoring.py`

\- axe-core automated accessibility testing implemented through `bench/axe\_runner.py`



PACE enables controlled experiments comparing how different models and prompt strategies affect the accessibility quality of generated HTML components.

# What the Code Does



## app.py



This is the main Streamlit application.



It provides a graphical interface for:



\- configuring benchmark runs

\- editing registry files

\- launching experiments

\- visualizing results



The UI allows users to select models, prompt conditions, prompt variants, and component tasks.



---



## bench/run\_engine.py



This file contains the core benchmark execution engine.



It:



\- loops through combinations of models, prompt conditions, variants, and components

\- generates HTML using selected LLMs

\- saves generated outputs

\- runs accessibility scoring

\- records experiment results



Each run produces structured CSV output for later analysis.



---



## bench/scoring.py



This module implements the schema-based accessibility scoring system.



It:



\- parses generated HTML

\- derives expected accessibility properties from component metadata

\- evaluates accessibility checks such as:



&nbsp; - label associations  

&nbsp; - description associations  

&nbsp; - required state encoding  

&nbsp; - input type correctness  

&nbsp; - error message structure  

&nbsp; - group semantics  

&nbsp; - status announcements  



Each check produces a score between \*\*0 and 2\*\*, along with a rationale.



---



## bench/axe\_runner.py



This module runs axe-core automated accessibility testing.



It:



\- wraps HTML fragments in a minimal document

\- loads axe-core

\- executes configured accessibility rules

\- collects violations, passes, and incomplete checks

\- saves results for analysis



---



## bench/openrouter.py



Handles communication with language models through the OpenRouter API.



Responsibilities include:



\- sending prompts to models

\- receiving generated HTML

\- tracking token usage

\- estimating generation cost



---



## bench/registry.py



Handles loading and managing benchmark configuration files.



These include:



\- models

\- prompt conditions

\- variants

\- components

\- scoring definitions



---



## data JSON Files



These files define the benchmark configuration.



\### components.json



Defines the component tasks used in the benchmark.



Examples include:



\- text inputs  

\- email fields  

\- radio groups  

\- file uploads  

\- buttons  



---



### variants.json



Defines linguistic prompt variants used to test how wording affects model behavior.



Examples include:



\- action command  

\- short specification  

\- descriptive object phrase  

\- build instruction  

\- standards-oriented phrasing  



---



### prompt\_conditions.json



Defines system instructions applied to model generations.



Examples include:



\- baseline generation  

\- strict specification following  

\- front-end engineering persona  

\- accessibility-focused perspective  



---



### score.json



Defines the accessibility evaluation rubric.



It specifies:



\- available checks

\- scoring scale

\- component-check mappings



The scoring logic itself is implemented in `bench/scoring.py`.



---



# How the Benchmark Works



For each benchmark trial, PACE:



1\. selects a model  

2\. selects a system instruction  

3\. selects a prompt variant  

4\. selects a component task  

5\. generates HTML from the model  

6\. saves the output  

7\. runs schema-based accessibility scoring  

8\. runs axe-core evaluation  

9\. records results in CSV files  



This creates a fully crossed experimental design.

# Installation



\## 1 Clone the repository



git clone https://github.com/momentine/pace.git  

cd pace



---



## 2 Create a virtual environment



Windows:



python -m venv .venv  

.venv\\Scripts\\activate  



macOS or Linux:



python -m venv .venv  

source .venv/bin/activate  



---



## 3 Install Python dependencies



Run the following command:



pip install streamlit pandas numpy altair requests python-dotenv beautifulsoup4 lxml playwright



---



## 4 Install Playwright browsers



playwright install



---



## 5 Install axe-core



PACE expects `axe.min.js` to be available.



Install it with:



npm install axe-core



This installs the file at:



node\_modules/axe-core/axe.min.js

&nbsp;  - variants

&nbsp;  - components

3\. Go to the Run tab.

&nbsp;  - models  

&nbsp;  - prompt conditions  

&nbsp;  - variants  

&nbsp;  - components

## results.csv



Contains one row per generation trial.



Fields include:



\- model  

\- system instruction

\- variant  

\- component  

\- repetition index  

\- token usage  

\- cost  

\- schema score  

\- axe violations  



---



## per\_check.csv



Contains detailed results for each accessibility check.



Each row includes:



\- check ID  

\- score  

\- explanation  

\- associated output file

# Notes



\- PACE focuses on HTML-only interface generation.

\- Schema scoring and axe evaluation measure different accessibility properties.

\- Generated runs are excluded from version control.



---



# Description



PACE is a controlled benchmarking framework for evaluating accessibility in LLM-generated HTML through factorial variation of models, prompt instructions, linguistic prompt variants, and component tasks.
