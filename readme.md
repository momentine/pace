# PACE  
**Prompt Accessibility Controlled Evaluation**

PACE is a benchmarking framework for evaluating the accessibility of HTML generated by large language models (LLMs).

It supports controlled experiments across models, prompt conditions, prompt variants, and component tasks. Each generation is evaluated with two complementary methods:

- **Schema-based scoring** in `bench/scoring.py`
- **axe-core testing** in `bench/axe_runner.py`

---

## Overview

PACE helps you answer questions like these:

- Which models produce more accessible HTML?
- How much does prompt wording change accessibility outcomes?
- Which prompt conditions change accessibility outcomes?
- Which accessibility checks fail most often across component tasks?

## What Each Part Does

### `app.py`
Main Streamlit app.

Use it to:

- configure benchmark runs
- inspect and edit registries
- launch experiments
- view results

### `bench/run_engine.py`
Core execution engine.

It:

- loops through benchmark combinations
- sends prompts to models
- saves generated HTML
- runs schema scoring
- runs axe-core checks
- writes output files for analysis

### `bench/scoring.py`
Schema-based accessibility evaluator.

It parses generated HTML and scores checks such as:

- label association
- description association
- required state encoding
- input type correctness
- error structure
- group semantics
- status announcements

Each check returns:

- a score from **0 to 2**
- a short rationale

### `bench/axe_runner.py`
axe-core integration for automated accessibility testing.

It:

- wraps fragments in a minimal document
- loads `axe.min.js`
- runs configured rules
- captures violations, passes, and incomplete checks

### `bench/openrouter.py`
OpenRouter integration.

It handles:

- prompt submission
- model responses
- token usage
- cost estimates

### `bench/registry.py`
Registry loader and manager.

It works with:

- models
- prompt conditions
- variants
- components
- scoring definitions

## Configuration Files

### `data/components.json`
Defines benchmark tasks.

Examples:

- text inputs
- email fields
- radio groups
- file uploads
- buttons

### `data/variants.json`
Defines prompt wording variants.

Examples:

- action command
- short specification
- descriptive object phrase
- build instruction
- standards-oriented phrasing

### `data/prompt_conditions.json`
Defines higher-level prompting conditions.

Examples:

- baseline
- strict specification
- front-end engineering persona
- accessibility-focused perspective

### `data/models.json`
Stores the OpenRouter model catalog used by the app.

This file is generated by a script, not edited by hand.

### `data/score.json`
Defines the scoring rubric and check mappings.

The scoring logic itself lives in `bench/scoring.py`.

---

## How PACE Runs

For each trial, PACE does the following:

1. selects a model
2. selects a prompt condition
3. selects a prompt variant
4. selects a component task
5. generates HTML
6. saves the output
7. runs schema-based scoring
8. runs axe-core evaluation
9. records the results

This creates a fully crossed benchmark design.

```bash
pip install streamlit pandas numpy altair requests python-dotenv beautifulsoup4 lxml playwright
```

## Fetch the Latest OpenRouter Models

Before running the app, refresh the model list from OpenRouter so `data/models.json` stays current.

### Save this script as `scripts/fetch_openrouter_models.py`

```python
from __future__ import annotations

import json
from pathlib import Path

import requests

OUT = Path("data/models.json")
OUT.parent.mkdir(parents=True, exist_ok=True)

URL = "https://openrouter.ai/api/v1/models"

def main() -> None:
    r = requests.get(URL, timeout=30)
    r.raise_for_status()
    data = r.json()

    # Optional: keep only fields you actually use (smaller + faster UI).
    slim = {"data": []}
    for m in data.get("data", []):
        slim["data"].append(
            {
                "id": m.get("id", ""),
                "name": m.get("name", ""),
                "canonical_slug": m.get("canonical_slug", ""),
                "context_length": m.get("context_length", None),
                "architecture": (m.get("architecture", {}) or {}),
                "pricing": (m.get("pricing", {}) or {}),
                "top_provider": (m.get("top_provider", {}) or {}),
                "supported_parameters": m.get("supported_parameters", []) or [],
                "default_parameters": m.get("default_parameters", {}) or {},
                "expiration_date": m.get("expiration_date", None),
            }
        )

    OUT.write_text(json.dumps(slim, indent=2, ensure_ascii=False) + "\n", encoding="utf-8")
    print(f"Saved {len(slim['data'])} models -> {OUT}")

if __name__ == "__main__":
    main()
```

### Run the script

```bash
python scripts/fetch_openrouter_models.py
```

Expected output:

```text
Saved 123 models -> data/models.json
```

You should run this again any time you want to refresh the available OpenRouter catalog.

## Run a Benchmark

1. Open the app.
2. Go to the **Settings** panel.
3. Inspect or edit the registries.
4. Go to the **Run** tab.
5. Select:
   - models
   - prompt conditions
   - variants
   - components
   - repetition count
6. Click **Run Benchmark**.

## Output

Each run creates a directory under:

```text
output/_runs/
```

Example:

```text
output/_runs/run_123456/
```

A run directory may include:

```text
results.csv
per_check.csv
generated HTML files
axe JSON results
```

### `results.csv`
One row per generation trial.

Typical fields:

- model
- system instruction
- variant
- component
- repetition index
- token usage
- cost
- schema score
- axe violations

### `per_check.csv`
Detailed accessibility check results.

Typical fields:

- check ID
- score
- explanation
- associated output file

## Notes

- PACE currently focuses on HTML-only interface generation.
- Schema scoring and axe-core capture different accessibility signals.
- Generated runs are stored in `output/_runs/`.
- Those run artifacts are typically excluded from version control.
- `data/models.json` should be treated as generated data.
